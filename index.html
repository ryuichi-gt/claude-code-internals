Technical Deep Dive

Overview

Claude Code とは何か

Claude Code は Anthropic が公式に提供する CLI（コマンドラインインターフェース）ツール。ターミナル上で自然言語の指示を与えると、ローカルマシン上でコードの読み書き・コマンド実行・デバッグを自律的に繰り返し、ソフトウェアを構築する。

その本質は 「LLM による推論」と「ローカルPC上でのツール実行」を高速ループさせるエージェントシステム。魔法のように見えるが、使われている技術は一つ一つ分解すると理解可能なものばかりだ。

Core Architecture

エージェントループ：全体のデータフロー

Claude Code の動作原理は「推論 → ツール実行 → 結果を推論に返す」ループの反復。1つのタスクで数十〜数百回このサイクルが回る。

① ユーザー入力
ターミナルに日本語テキストを入力。例：「Next.js でブログサイトを作って」
stdin / Ink UI
UTF-8 テキストをそのまま送信（翻訳・変換なし）
② API リクエスト組み立て
CLI が以下を1つの JSON リクエストにまとめて Anthropic Messages API へ POST 送信する
JSON Messages API
HTTPS POST → api.anthropic.com/v1/messages
③ Claude（LLM）推論
入力全体を解析し、「次に何をすべきか」を決定。テキスト応答 or ツール呼び出しを生成
Claude claude-opus-4-6
Server-Sent Events（SSE）でストリーミング返却
④ レスポンス解析・分岐
CLI がレスポンスを解析：
A. テキストのみ → ターミナルに表示して完了
B. tool_use ブロックあり → ⑤ へ進む
TypeScript パーサー
tool_use が含まれる場合
⑤ 許可チェック & ツール実行
許可モードに応じてユーザーに確認 → 許可されればローカルでツールを実行
例：ファイル書き込み、npm install、git commit 等
child_process Bash / fs
実行結果を tool_result として次の API リクエストに含める
⑥ ループ継続（② に戻る）
ツールの実行結果を含めて再度 API を呼び出す。Claude が「もうツールは不要」と判断するまで②〜⑥が繰り返される。
1タスクあたり平均 20〜100+ 回のAPIコールが発生する。
Loop
API Protocol

API リクエストの構造

Claude Code が毎回 Anthropic API に送信するリクエストの実体。これが「コンテキストウィンドウ」と呼ばれるもの。

// POST https://api.anthropic.com/v1/messages { "model": "claude-opus-4-6", "max_tokens": 16384, "stream": true, // ── System Prompt ── "system": [ "You are Claude Code, Anthropic's official CLI...", "// + CLAUDE.md の内容", "// + メモリファイルの内容", "// + 環境情報（OS, shell, cwd, git状態）" ], // ── Tools（使えるツール一覧）── "tools": [ { "name": "Bash", "description": "Execute a bash command...", "input_schema": { "type": "object", "properties": { "command": { "type": "string" }, "timeout": { "type": "number" } } } }, { "name": "Read", /* ... */ }, { "name": "Write", /* ... */ }, { "name": "Edit", /* ... */ }, { "name": "Glob", /* ... */ }, { "name": "Grep", /* ... */ }, { "name": "Agent", /* サブエージェント起動 */ } ], // ── Messages（会話履歴）── "messages": [ { "role": "user", "content": "Next.jsでブログサイトを作って" }, { "role": "assistant", "content": [ { "type": "tool_use", "name": "Bash", "input": { "command": "npx create-next-app@latest blog" } } ] }, { "role": "user", // ← CLI がツール結果を「user」ロールで返す "content": [ { "type": "tool_result", "tool_use_id": "toolu_xxx", "content": "Success: Created blog/ directory..." } ] } // ... このパターンが何十回も繰り返される ] }
重要ポイント： 日本語の入力は一切変換されない。UTF-8 テキストがそのまま messages 配列に入り、API に送信される。Claude が多言語を理解できるため、翻訳レイヤーは不要。
Tool Use / Function Calling

ツール実行の仕組み

Claude Code が「万能」に見える理由の核心がこれ。Claude 自体はテキストしか出力できないが、特定の JSON フォーマットを出力することで「ツールを呼び出したい」という意思を表明し、CLI がそれを実際に実行する。

Claude の応答に含まれる tool_use ブロック（API が返す JSON）

{ "type": "tool_use", "id": "toolu_01ABC...", "name": "Write", "input": { "file_path": "/Users/you/blog/app/page.tsx", "content": "export default function Home() {\n return <h1>My Blog</h1>\n}" } }

CLI 側の処理（TypeScript）

// 概念的な擬似コード（実際のコードを簡略化） async function executeToolCall(toolUse: ToolUseBlock) { switch (toolUse.name) { case 'Bash': // Node.js の child_process.spawn でシェルコマンドを実行 return spawn('bash', ['-c', toolUse.input.command]); case 'Read': // fs.readFile でファイルを読み取り return fs.readFile(toolUse.input.file_path, 'utf-8'); case 'Write': // fs.writeFile でファイルを書き込み return fs.writeFile(toolUse.input.file_path, toolUse.input.content); case 'Edit': // ファイルを読み → 文字列置換 → 書き戻し const file = await fs.readFile(path); return fs.writeFile(path, file.replace(old, new)); case 'Grep': // ripgrep (rg) コマンドをサブプロセスとして実行 return spawn('rg', [pattern, path, ...flags]); case 'Agent': // 新しい Claude API セッションをサブプロセスとして起動 return spawnSubAgent(toolUse.input.prompt); } }
ツール名裏で動く技術実装の実体
Bashchild_process.spawnユーザーのシェル（zsh/bash）でコマンドをサブプロセス実行。stdin/stdout/stderr をキャプチャ
Readfs.readFileNode.js のファイルシステムAPI。バイナリ（画像）は Base64 エンコードして返却
Writefs.writeFileファイルの新規作成 or 上書き。ディレクトリが無ければ mkdirp で再帰作成
Edit文字列検索 + 置換old_string の完全一致を探して new_string に置換。一意性を保証するため重複があればエラー
Globfast-glob ライブラリファイルパターンマッチング（例: **/*.tsx）。OS の find より高速
Grepripgrep（rg）Rust 製の超高速テキスト検索ツール。正規表現対応、.gitignore 自動除外
Agent再帰的 API セッション新しい Messages API セッションを起動し、独立したコンテキストでサブタスクを実行
Context Window

コンテキストウィンドウ管理

Claude の入力には上限（200K トークン = 約15万語）がある。長い作業ではこれを超えるため、Claude Code は以下の戦略でコンテキストを管理する。

自動コンパクション（Auto-Compaction）

コンテキストが上限に近づくと、CLI が過去の会話を自動要約し、古いメッセージを圧縮して新しい情報のためのスペースを確保する。ユーザーからは「会話が途切れない」ように見えるが、裏では古いやり取りが要約テキストに置き換わっている。

サブエージェント（Agent ツール）

メインのコンテキストウィンドウを汚さないために、独立した調査タスクは サブエージェント に委任できる。サブエージェントは独自のコンテキストウィンドウを持ち、結果だけをメインに返す。人間のチームで「調べておいて」と依頼するのと同じ構造。

メモリシステム（永続記憶）

会話をまたいで保持すべき情報は ~/.claude/projects/*/memory/ に Markdown ファイルとして保存。次回のセッション開始時にシステムプロンプトに注入される。これにより「前回の続き」を覚えていられる。

Security

許可モデルとセキュリティ

Claude Code はユーザーのローカルマシンで任意のコマンドを実行できるため、多層的な許可チェックが組み込まれている。

全自動モード
--dangerously-skip-permissions フラグで起動。すべてのツール呼び出しを自動承認。開発環境専用
選択承認モード
settings.json の allowedTools で許可するツールを個別指定。それ以外は毎回ユーザーに確認ダイアログを表示
都度確認モード
デフォルト。すべてのツール呼び出しで Ink UI が許可ダイアログを表示し、ユーザーが Accept / Reject を選択
Hooks
ツール実行の前後に任意のシェルスクリプトを実行できるフック機構。組織ポリシーの強制や監査ログに利用
Streaming

ストリーミング応答の仕組み

Claude Code でテキストがリアルタイムに流れるように表示されるのは、Server-Sent Events（SSE） というHTTPストリーミングプロトコルのおかげ。

// API が返す SSE ストリーム（実際の通信内容） event: content_block_delta data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"Next"}} event: content_block_delta data: {"type":"content_block_delta","delta":{"type":"text_delta","text":".js"}} event: content_block_delta data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"で"}} event: content_block_delta data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"ブログ"}} // → CLI はこれを受信するたびにターミナルに1文字ずつ描画

通常のHTTPリクエストは「全部生成してから返す」が、SSE では生成途中のトークンを1つずつリアルタイムに送信する。CLI 側は EventSource パーサーでこれを受け取り、Ink の React コンポーネントを逐次更新する。

End-to-End Example

具体例：「ブログを作って」の裏側

ユーザーが「Next.js でブログサイトを作って」と入力してから完成するまでに、裏側で何が起きているかの具体的なシーケンス。

#Claude の判断呼び出すツール裏側の処理
1 「まずプロジェクトを作ろう」 Bash child_process.spawn('bash', ['-c', 'npx create-next-app@latest blog --ts --app'])
2 「作成されたか確認」 Glob fast-glob('blog/**/*') でファイル一覧を取得
3 「トップページを書き換え」 Write fs.writeFile('blog/app/page.tsx', '...')
4 「記事データの型を定義」 Write fs.writeFile('blog/types/post.ts', '...')
5 「記事一覧コンポーネント作成」 Write fs.writeFile('blog/components/PostList.tsx', '...')
6 「CSSを調整」 Edit fs.readFile → String.replace(old, new) → fs.writeFile
7 「ビルドして動くか確認」 Bash child_process.spawn('bash', ['-c', 'cd blog && npm run build'])
8 「ビルドエラーが出た。修正」 Edit エラー出力を解析 → 該当ファイルを修正
9 「再ビルド」 Bash child_process.spawn('bash', ['-c', 'cd blog && npm run build'])
10 「成功。ユーザーに報告」 — テキスト応答のみ → ターミナルに表示して終了
ポイント： 各ステップの「Claude の判断」は、前のステップの結果を含む会話全体を入力として行われる。ステップ8の「エラーを修正」は、ステップ7で得たエラーログを読んで推論した結果。事前にプログラムされたルールではなく、毎回その場でLLMが判断している。
Misconceptions

よくある誤解

× 「裏でコードを実行するAI」

Claude 自体はコードを実行できない。テキストを生成するだけ。CLI がそのテキスト（ツール呼び出しJSON）を解釈して、ユーザーのPC上で実行する。実行環境はあくまでユーザーのローカルマシン。

× 「日本語を英語に翻訳してから処理している」

翻訳は一切していない。Claude の学習データには日本語テキストが大量に含まれており、日本語をそのまま理解する。内部的にはトークナイザーが日本語をサブワードトークンに分割するが、これは「翻訳」ではなく数値表現への変換。

× 「テンプレートやスニペットの組み合わせ」

事前定義されたテンプレートは使っていない。毎回、Transformer ニューラルネットワークが次のトークンを確率的に予測して生成する。同じ指示でも毎回微妙に異なるコードが生成されるのはこのため。
